# Overview
Kafe Web is a full-featured café management frontend built with Next.js 16 (App Router). It serves three distinct user types — customers, baristas, and admins — through a single application with role-based views. The system solves the problem of managing a café's entire operation in one place: customers browse a product catalog and place orders without needing an account; baristas manage an order queue in real time; and admins control the full business — products, categories, users, inventory, and analytics.

The frontend connects to an existing NestJS REST API. All business logic and data persistence live in the backend. The frontend's job is to expose that API surface through clear, role-appropriate UIs.

# Core Features

## 1. Public Product Catalog & Order Creation
- **What it does:** Displays all available products organized by category. Customers add items to a cart and submit an order (anonymously or as an authenticated client).
- **Why it's important:** This is the primary revenue-generating flow. It must work without an account.
- **How it works:** Fetches categories (GET /categories) and products (GET /products?categoryId=) as Server Components. Cart state is managed client-side with React Context + sessionStorage. Order submission calls POST /orders.

## 2. Authentication
- **What it does:** Login flow using email + password. JWT stored in an httpOnly cookie. Middleware protects role-gated routes.
- **Why it's important:** Barista and admin interfaces require authenticated sessions. Client accounts enable order history.
- **How it works:** Login page posts to a Next.js route handler (POST /api/auth/login) which calls POST /auth/login on the NestJS API, sets an httpOnly cookie with the JWT, then redirects. Middleware decodes the JWT payload (no signature verification needed — backend enforces auth) to determine role and protect routes.

## 3. Client Order History
- **What it does:** Authenticated clients can view their past orders with status.
- **Why it's important:** Builds trust and gives customers visibility into their order progression.
- **How it works:** GET /orders/me fetched as a Server Component. Status displayed with color-coded badges.

## 4. Barista Order Queue
- **What it does:** Baristas see a live-updating queue of active orders (RECEIVED and IN_PREPARATION). They advance order status with a single button click.
- **Why it's important:** The operational heart of the café. Must be real-time.
- **How it works:** Client Component polling GET /orders/queue every 15 seconds. Status updates call PATCH /orders/:id/status. Inventory alert banner at top if GET /inventory/alerts returns any items.

## 5. Admin Dashboard
- **What it does:** Displays sales summary, top-selling products, and peak order hours for a configurable date range.
- **Why it's important:** Gives the owner actionable business intelligence.
- **How it works:** Three parallel Server Component fetches to /dashboard/summary, /dashboard/top-products, and /dashboard/peak-hours. Date range controlled via URL params. Peak hours rendered as native SVG bar chart (no chart library).

## 6. Admin Product & Category Management
- **What it does:** Full CRUD for products and categories. Toggle product availability. Manage product ingredient recipes.
- **Why it's important:** Admins need to update the menu without developer intervention.
- **How it works:** Server-rendered list pages with client-side modals for create/edit/delete confirmations. API calls go to /products and /categories endpoints.

## 7. Admin User Management
- **What it does:** List, create, edit, and deactivate users. Assign roles (ADMIN, BARISTA, CLIENT).
- **Why it's important:** Admins control who has access to what.
- **How it works:** CRUD against /users endpoints. Role assignment via dropdown.

## 8. Admin Inventory Management
- **What it does:** List ingredients with stock levels, restock ingredients, view stock movement history.
- **Why it's important:** Prevents ingredient shortages that would make menu items unavailable.
- **How it works:** Server-rendered list from GET /inventory. Restock form calls POST /inventory/:id/restock. Movements table from GET /inventory/movements with date + ingredient filters.

# User Experience

## User Personas

**Anonymous Customer:** Walks up to a kiosk or scans a QR code. Wants to see the menu, pick items, and submit an order quickly. No desire to create an account.

**Client (Registered Customer):** Regular customer who logs in to track their order history. Expects their name pre-filled on orders.

**Barista:** Staff member working the counter. Needs a clear, fast interface to manage their queue. Always authenticated. Views on tablet or desktop.

**Admin / Owner:** Manages the business. Updates the menu seasonally, monitors sales, manages stock. Views on desktop.

## Key User Flows

**Customer order flow:**
1. Land on / -> see product grid organized by category tabs
2. Click product -> add to cart (slide-in cart sidebar or bottom bar on mobile)
3. Click "Place Order" -> fill in name + optional notes -> confirm -> see order confirmation with order ID

**Barista queue flow:**
1. Login -> redirect to /barista/queue
2. See cards for each active order (RECEIVED -> IN_PREPARATION -> READY)
3. Click "Start Preparation" -> status advances to IN_PREPARATION
4. Click "Ready" -> status advances to READY
5. Click "Delivered" -> order leaves the queue

**Admin menu update flow:**
1. Login -> redirect to /admin/dashboard
2. Navigate to /admin/products -> click "New Product"
3. Fill form (name, category, price, image URL, description) -> Save
4. Product appears on public catalog immediately

## UI/UX Considerations
- Mobile-first layout for public catalog (customers on phones)
- Tablet-optimized for barista queue (fixed at counter)
- Desktop-first for admin panel (keyboard + mouse)
- Color-coded order status badges consistent across all views:
  - RECEIVED: blue, IN_PREPARATION: amber, READY: green, DELIVERED: gray, CANCELLED: red
- Toast notifications for all mutations (success + error)
- Skeleton loaders for all async data
- Confirm modal before any destructive action (delete, cancel order)
- Unsaved changes warning on admin forms (beforeunload)

## Authentication Strategy
- JWT stored in httpOnly cookie named `kafe_token`
- Next.js route handler POST /api/auth/login calls NestJS, receives token, sets cookie via Set-Cookie header
- middleware.ts runs at the edge, decodes JWT payload (no crypto verification), reads `role` field, enforces:
  - /barista/* -> requires BARISTA or ADMIN
  - /admin/* -> requires ADMIN
  - /orders/me -> requires any authenticated role
  - /login -> redirects to dashboard if already authenticated
- Server Components read the token from next/headers cookies() and forward it as Authorization: Bearer
- Client Components never call NestJS directly; they call Next.js route handlers

## API Client (lib/api/client.ts)
```typescript
// Server-side: reads token from next/headers
// Client-side: token forwarded by route handlers
export async function apiRequest<T>(
  path: string,
  options?: RequestInit & { token?: string }
): Promise<T>
```
All API modules export typed functions using apiRequest.
Example: getProducts(query: ListProductsQuery): Promise<PaginatedResponse<Product>>

## Server vs Client Components
- Catalog page (/): Server — SEO + initial load performance
- CategoryTabs: Client — interactive filter selection
- CartSidebar: Client — local state
- Barista queue: Client — polling
- Dashboard: Server — no interactivity; parallel data fetch
- DateRangePicker: Client — user interaction
- Admin list pages: Server — simple data display
- Admin create/edit forms: Client — form state + validation

## Route Protection (middleware.ts)
```typescript
const PROTECTED_ROUTES = [
  { pattern: /^\/admin/, roles: ['ADMIN'] },
  { pattern: /^\/barista/, roles: ['BARISTA', 'ADMIN'] },
  { pattern: /^\/orders\/me/, roles: ['CLIENT', 'BARISTA', 'ADMIN'] },
];
```
JWT payload decoded with a pure base64 decode (no library, edge-safe).
Unauthenticated users redirected to /login. Wrong role redirected to /.

# Risks and Mitigations

## Risk 1: JWT decode at edge without a library
The middleware runs in edge runtime. No Node.js crypto, no jose library installed.
Mitigation: Decode base64url payload only — no signature verification. Backend enforces auth on every API call; routing redirects are UX only.
```typescript
function decodeJwtPayload(token: string): { role?: string } | null {
  try {
    return JSON.parse(atob(token.split('.')[1].replace(/-/g, '+').replace(/_/g, '/')));
  } catch { return null; }
}
```

## Risk 2: Server Component token forwarding
Server Components cannot access httpOnly cookies via document.cookie.
Mitigation: Use cookies() from next/headers in Server Components. Client Components use Next.js route handlers as proxies — never call NestJS directly.

## Risk 3: Cart state lost on hard refresh
In-memory CartContext resets on browser refresh.
Mitigation: Sync cart state to sessionStorage on every change. Read from sessionStorage on mount with a try/catch guard for SSR safety.

## Risk 4: Polling stale closures
setInterval captures stale React state via closure.
Mitigation: usePolling uses useRef to hold the latest callback version. The interval fires ref.current() instead of the captured callback.

## Risk 5: API URL misconfiguration
Missing NEXT_PUBLIC_API_URL causes all API calls to fail silently.
Mitigation: client.ts throws an error at module load time if the variable is undefined — fails loudly and immediately.

## Risk 6: Admin form data loss on navigation
Clicking away from unsaved admin form loses all input.
Mitigation: useFormDirty hook tracks dirty state. Attaches beforeunload event listener when dirty. Triggers confirm() on Next.js client-side navigation.

# Appendix

## A. Order Status State Machine
```
RECEIVED -> IN_PREPARATION -> READY -> DELIVERED
    |              |             |
    +--------------+-------------+-> CANCELLED
```
Next status map in StatusButton:
```typescript
const NEXT_STATUS: Partial<Record<OrderStatus, OrderStatus>> = {
  RECEIVED: 'IN_PREPARATION',
  IN_PREPARATION: 'READY',
  READY: 'DELIVERED',
};
```
CANCELLED is always available as a destructive secondary action.
DELIVERED and CANCELLED have no next state — advance button is hidden.

## B. Environment Variables
- NEXT_PUBLIC_API_URL — NestJS API base URL (e.g., http://localhost:3001)
No other secrets needed in the frontend. JWT verification is backend-only.

## D. Pagination Convention
All list endpoints use ?page=1&limit=20. The Pagination component reads/writes the page URL param. limit is a constant per page (20 for most lists, 10 for top products dashboard widget).

## E. Date Handling (no library)
- Display: new Date(isoString).toLocaleDateString('en-US', { ... })
- API query param: date.toISOString().split('T')[0] -> YYYY-MM-DD
- Queue elapsed time: Math.floor((Date.now() - new Date(createdAt).getTime()) / 60000) minutes, recalculated each poll cycle

## F. Next.js Image Configuration
next/image used for all product images with explicit width and height props.
Fallback: /public/images/product-placeholder.svg when imageUrl is null.
next.config.ts must whitelist remote image domains via remotePatterns.

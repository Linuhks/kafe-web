# Architecture

`kafe-web` is a Next.js 16 App Router application. It consumes the `kafe-api` REST backend and exposes a role-based UI for three user types: ADMIN, BARISTA, and CLIENT.

---

## Stack

| Layer | Technology |
|-------|-----------|
| Framework | Next.js 16.2 (App Router) |
| Language | TypeScript 5 (strict, no `any`) |
| Styling | Tailwind 4 + Shadcn/Radix |
| State | React Query (TanStack v5) |
| Forms | react-hook-form + zod |
| Icons | lucide-react |
| API gen | Orval (React Query hooks from OpenAPI) |

---

## Rendering model

Pages are **Server Components by default**. Add `'use client'` only when the component needs interactivity, hooks, or browser APIs.

```
page.tsx           ← Server Component: fetch directly, no hooks
  └─ ClientWidget  ← 'use client': state, event handlers, React Query hooks
```

**Rule:** push `'use client'` as deep into the tree as possible. Keep pages and layouts as Server Components.

---

## Request flow

### Server Component fetching data

```
RSC (page.tsx)
  → serverFetch()          ← reads kafe_token cookie, adds Authorization header
    → apiFetch()           ← direct HTTP to backend (NEXT_PUBLIC_API_URL)
      → kafe-api
```

### Client Component fetching data

```
Client Component
  → React Query hook       ← generated by Orval
    → apiFetch()           ← base URL is empty (client-side)
      → /api/v1/*          ← Next.js rewrite in next.config.ts
        → kafe-api
```

Both paths share the same `apiFetch` function. The difference is the base URL: server-side uses the full `NEXT_PUBLIC_API_URL`, client-side uses an empty string and relies on the rewrite.

## Authentication

- **Cookie**: `kafe_token` — httpOnly, secure, sameSite: lax. Set by the `/api/auth/login` route handler after the backend validates credentials.
- **Token format**: Bearer JWT signed by the backend.
- **Session verification**: `proxy.ts` (Next.js middleware) calls `GET /api/auth/get-session` on every non-asset request to validate the token and read the user's role.
- **Logout**: `POST /api/auth/logout` deletes the cookie server-side.

---

## Middleware (proxy.ts)

`proxy.ts` runs before every page render and enforces access control:

| Path | Required role |
|------|--------------|
| `/login` | unauthenticated (redirects if already logged in) |
| `/admin/*` | `ADMIN` |
| `/barista/*` | `BARISTA` or `ADMIN` |
| `/orders/me` | any authenticated role |
| everything else | public |

Do not replicate this logic in pages or layouts — the middleware is the single source of truth for access control.

---

## Security headers

Configured in `next.config.ts` and applied globally:

- `Content-Security-Policy` — restricts script, style, image, connect origins
- `X-Frame-Options: DENY`
- `X-Content-Type-Options: nosniff`
- `Referrer-Policy: strict-origin-when-cross-origin`
- `Permissions-Policy` — disables camera, microphone, geolocation
